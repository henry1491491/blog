Python Django 資料庫（Schema）設定

  • model 的欄位
  • 欄位裡面的設定

model 的欄位

  • CharField 短中長度、長度固定的字串，必須設定最大長度

  • TextField 任意長度字串，最好設定最大長度限制（但只會作用在表單，資料庫不會去做限制）

  • IntegerField 儲存整數的欄位，在表格階段被驗證

  • DateField and DateTimeField 時間設定

目前還沒使用過

  • EmailField 儲存及驗證 E-mail 地址

  • FileField and ImageField 儲存上傳檔案跟圖檔（可以設定檔案或是照片的位置）

  • AutoField 一個自動增加的整數欄位。若無指定，會自動為你增加主鍵（Primary Key）

  • ForeignKey 外鍵，可對應到其它 models

  • ManyToManyField

還沒用過，等用過再來紀錄

更多類型參考：Model field reference | Django documentation | Django

欄位裡面的設定

可以設定在 models.CharField() 方法裡的參數：

  • help_text 創建一個 html 表單的 Label

  • verbose_name 用來給人閱讀的表單標籤名稱。如果沒設定會用 help_text 作為預設

  • default 每新增一筆紀錄，會自動創建的 field 的預設值

  • null

還沒使用，未來補充

  • blank 允許空格，與上方 null 設定有關

還沒使用，未來補充

  • choices

還沒使用，未來補充

  • primary_key 將這個欄位設為主鍵，如果沒有，Django 會自動幫你增加一個欄位為主鍵

更多設定參考：Model field reference | Django documentation | Django

Python Django 專案資料夾結構

  • init.py：當 Python 讀到這個檔案，便會將該層資料夾視為一個 Python Package
  • settings.py：設定專案的地方。註冊 applicatios、存放靜態檔案的位置、資料庫細部的設定等等
  • urls.py：設定 URL 對應 View 的地方（路由對應頁面，或對應到特定 application）
  • wsgi.py：與 webserver 溝通的設定
  • asgi.py：Python 網站與 server 進行異步溝通的標準，與 wsgi 差別在於它同時提供同步與非同步與 Python 網站的溝通，是 wsgi 的向下相容，支援多個 server 跟 appications 框架

上述部分設定還沒有細究，僅僅先有個概念

為了讓不同 Python 專案之間環境乾淨，或是避免影響到全域環境，通常很多人會建立虛擬環境來進行開發

pip3 安裝虛擬環境（因為使用 Python3, macOS）

打開 zsh 配置檔（因為我的 shell 使用的是 zsh）

加上以下幾行在結尾

  • VIRTUALENVWRAPPER_PYTHON 對應到 python3 的位置
  • /usr/local/bin/virtualenvwrapper.sh 這行也要確定是否該路徑下存放 virtualenvwrapper.sh 是對的
  • 以上兩者確認路徑方式可透過

設定完成後，重置設定

然後我們可以在終端機看到提示開頭的 () 裡面就是剛剛創建的虛擬環境

年度回顧

2020 年度回顧

這是 27 年來第一次進行年度回顧，期許這個儀式可以持續下去。此篇回顧是第 3 版。

一直以來，頭腦靜不下來的我總會想到一些重複出現的內容。最常想到的就是「相較那些天才、那些世俗成功的人來說，自己還不夠努力、不夠聰明。」。「完美主義」一直是個捆擾已久的課題，而身為平凡人類的我們，因為追求不成，感受到的總是絕望居多。

無法精準地執行自我設下的目標、有太多未知與突如其來的事件影響著，加上身為人類有太多情感方面的弱點，也讓我曾一度想著要躲開一些人際關係，過著自給自足式的生活。其實現在還是這麼想著，只是身為凡人的我，還是會有需要情感支持的時候。

在抽完 2021 流年牌，提醒了我不要過度活在一個不切實際的夢想中。踏實地築夢，我想可以用來當作座右銘。這又讓我想到，2019 年在保安宮抽到的靈籤，就一直貼在我書桌前提醒著自己：

「真金經火煉千回，此物生成七寶魁。分明骨骼非常在，誰肯移來就炭灰。」

也許我對自我成長的追求，永遠不夠吧！可以說是到了一個貪婪的程度。但這或許是件好事吧？純粹只是心態上，而實際上自己執行的成果、方向是否正確，那又是一個需要時間驗證的事了。

但確確實實地，我感謝這樣的自己。就像電影裡那些歷經各種試煉、各種打擊還倒不下的鬥士，我慶幸自己始終有著與之看齊的心。

思索回顧這件事的意義

  • 人常常高估自己能力、對於事情的健忘程度，提醒自己一些未完成的目標
  • 人的生命短暫，每分每秒極其珍貴
  • 警惕自己不要當個「不斷做同樣的事，卻期望有所不同」的人
  • 警惕自己不要當個隨隨便便過日子的人（必須要活得精彩）

Asset Folders

全域的資產資料夾

在 source 資料夾內，不屬於發佈文章類型的檔案稱為 Assets。假如你只是要放幾張照片在一個 Hexo 專案裡，那麼最簡單的方式就是將它們放入 source/images 資料夾。然後，你可以透過 ![](/images/image.jpg) 像是這樣的方式存取到它們。

發佈資產資料夾

針對那些預期會經常提供照片或其他資料，或是偏好將他們的資產以「一篇發文對應一個資料夾」的形式存放的使用者們，Hexo 也提供一個更有組織的方式來管理這些資產。這涉及到更多，但資產處理可透過將 _config.yml 裡 post_asset_folder 的設定為 true 被非常輕易地啟動。

隨著啟用資產資料夾管理，Hexo 將會在每次你透過 hexo new [layout] <title> 指令創建新發文時，建立一個資料夾。這個資料夾名稱將會基於這個發文所產生與發文的 markdown 檔案一樣的名稱。將所有相關於這篇發文的資產放進去相對應的資料夾，且你能夠透過相對路徑的方式去引用它們，創建一個更加容易且便利的工作流。

針對相對路徑引用的標記套件

透過一般的 markdown 語法及相對路徑來引用相片或是其他資產可能導致無法正確顯示在 archives 或是 index 頁面。在 Hexo 2，在社群裡有套件被發明用來解決這個問題。然而，僅有少數新的標記套件被添加到 Hexo 3 的發佈。以下在發文中引用你的資產的方式將會更簡單：

舉例來說，在啟用資產資料夾的情況下，你放了一張叫做 example.jpg 的相片進入資產資料夾。假如你透過 ![](example.jpg) 的 markdown 語法，使用相對路徑來引用它，這將不會在 index 頁面顯示。（然而用在發文本身是預期可以成功的）。

因此針對引用相片更為正確的用法將會是透過標記套件語法，而非 markdown。
